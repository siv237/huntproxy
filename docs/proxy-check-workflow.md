# Процесс проверки и работы прокси в huntproxy

## Цель документа

Описать простыми шагами, как huntproxy превращает скачанные из интернета адреса прокси в рабочий список, через который пользователь может безопасно выходить в интернет.

## Главное правило

Прокси считается пригодным, только если через него может идти **HTTPS-трафик**. Для HTTP-прокси это значит обязательную поддержку `CONNECT` (туннель HTTPS). SOCKS-прокси пригодны всегда, если проходят SOCKS-рукопожатие. Простые HTTP-прокси без CONNECT нам не нужны.

---

## 1. Сбор адресов

Система скачивает списки публичных прокси из настроенных источников (обычно репозитории на GitHub). Из каждого списка извлекаются пары `IP:порт`.

```text
 Запуск huntproxy
       │
       ▼
 Скачивание списков прокси
       │
       ▼
 Извлечение адресов IP:порт
       │
       ▼
 ┌──────────────┐
 │ Адрес        │────Нет────▶ Пропустить
 │ валиден?     │
 └──────┬───────┘
        │ Да
        ▼
 Добавить в общий пул
```

---

## 2. Первичная проверка (Hunt)

Каждый адрес проверяется в двух потоках одновременно: по HTTP и по SSL (TLS). Прокси считается рабочим, если хотя бы одна проверка прошла, **и** при этом выполняется условие туннелирования HTTPS.

Проверки:
- **HTTP-проверка** — порт отвечает, HTTP-запрос проходит, определяется страна выхода.
- **SSL-проверка** — порт принимает TLS-соединение (валидность сертификата не проверяется).
- **CONNECT** — умеет ли прокси строить HTTPS-туннель.
- **MITM** — если CONNECT прошёл, через `curl` проверяется, не подсовывает ли прокси поддельный сертификат.

```text
                          Прокси из пула
                                │
                                ▼
               ┌────────────────────────────────┐
               │ Проверяем параллельно два пути │
               │ 1. HTTP                        │
               │ 2. SSL (TLS-соединение)        │
               └────────────────┬───────────────┘
                                │
               ┌────────────────┴────────────────┐
               │                                 │
               ▼                                 ▼
  ┌────────────────────────┐        ┌────────────────────────┐
  │ HTTP-проверка          │        │ SSL-проверка           │
  │ ─ порт отвечает?       │        │ ─ порт принимает TLS?  │
  │ ─ HTTP-запрос проходит?│        │ ─ получаем страну?     │
  │ ─ страна выхода?       │        │                        │
  └──────────┬─────────────┘        └──────────┬─────────────┘
             │                                 │
             └────────────────┬────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │ Хотя бы одна    │
                    │ проверка прошла?│
                    └────────┬────────┘
                             │
               ┌─────────────┴─────────────┐
              Нет                          Да
               │                           │
               ▼                           ▼
     Отметить как нерабочий      Прокси рабочий
                                       │
                                       ▼
                          ┌──────────────────────┐
                          │ HTTP-прокси?         │
                          │ Есть CONNECT?        │
                          │ (HTTPS-туннель)      │
                          └──────────┬───────────┘
                                     │
                       ┌─────────────┴─────────────┐
                      Нет                          Да
                       │                           │
                       ▼                           ▼
              HTTP-прокси без CONNECT      HTTPS-прокси, проверяем MITM
              НЕ НУЖЕН — откидываем                │
                                                   │
                                                   ▼
                                          ┌──────────────────┐
                                          │ MITM-подозрение? │
                                          │ (curl проверяет  │
                                          │ сертификат)      │
                                          └────────┬─────────┘
                                                   │
                                     ┌─────────────┴─────────────┐
                                    Да                           Нет
                                     │                           │
                                     ▼                           ▼
                             Пометить MITM              Всё в порядке
                             (прокси всё равно рабочий) (продолжаем)
```

## 3. Замер скорости

Каждый рабочий прокси замеряется на скорость. Способ зависит от его возможностей:

- **HTTPS-скорость** — если прокси поддерживает SSL и CONNECT. Открывается туннель `CONNECT host:443`, внутри него поднимается TLS и скачивается тестовый файл. Это самый точный замер.
- **HTTP-скорость** — если SSL/CONNECT нет, но прокси всё равно признан рабочим (например, SOCKS без SSL на порту, или HTTP-прокси с CONNECT но без SSL). В этом случае тестовый файл скачивается по HTTP.

```text
              Прокси рабочий
                    │
                    ▼
           ┌────────────────────┐
           │ Есть SSL + CONNECT?│
           │ (оба условия)      │
           └─────────┬──────────┘
                     │
       ┌─────────────┴─────────────┐
      Да                           Нет
       │                           │
       ▼                           ▼
┌─────────────────────┐   ┌─────────────────────┐
│ Открыть туннель     │   │ Обычный HTTP GET    │
│ CONNECT srv:443     │   │ к тестовому файлу   │
│ поднять TLS         │   │                     │
│ скачать по HTTPS    │   │                     │
└──────────┬──────────┘   └──────────┬──────────┘
           │                         │
           ▼                         ▼
    ┌──────────────┐           ┌──────────────┐
    │ Скорость     │           │ Скорость     │
    │ получена?    │           │ получена?    │
    └──────┬───────┘           └─────┬────────┘
           │                         │
  ┌────────┴────────┐        ┌───────┴───────┐
 Да                 Нет      Да             Нет
  │                 │        │               │
  ▼                 ▼        ▼               ▼
Записать        Ошибка  Записать        Ошибка
скорость        скорости скорость       скорости
```

## 4. Расчёт рейтинга (score)

Каждому рабочему прокси присваивается числовой рейтинг. Если последняя проверка не прошла (`last_status != ok`) или проверок не было, рейтинг сразу 0. Иначе рейтинг считается по формуле:

| Компонент | Как считается | Диапазон |
|-----------|---------------|----------|
| Успешность | `success_rate * 40` | 0 — 40 |
| Задержка | `(100 - latency_avg * 10) * 0.2`, но не меньше 0 | 0 — 20 |
| SSL | `+10` если `ssl_supported` | бонус |
| CONNECT | `+5` если `supports_connect` | бонус |
| MITM | `-30` если `mitm_suspect` | штраф |
| Скорость | `+min(50, speed_avg / 20)` | 0 — 50 |
| Ошибки скорости | `-15` за каждый фейл, но не более `-45` | штраф |
| IP-чёрный список | `result *= 0.75 ^ hits`, но не ниже 20% | множитель |
| Ручной чёрный список | рейтинг = 0 | исключение |
| Итог | не может быть отрицательным и не может превышать `100` | 0 — 100 |

```text
                    Результаты проверок
                            │
        ┌───────────────────┼───────────────────┐
        │                   │                   │
        ▼                   ▼                   ▼
  Успешность           Задержка             Скорость
  (0-40)               (0-20)               (0-50, -15 за фейл)
        │                   │                   │
        │                   │                   │
        │       SSL +10     │        MITM -30   │
        │       CONNECT +5  │                   │
        │           │       │           │       │
        └───────────┼───────┴───────────┼───────┘
                    │                   │
                    ▼                   ▼
           ┌────────────────────────────┐
           │ Сумма компонентов          │
           └──────────────┬─────────────┘
                          │
                          ▼
                  ┌───────────────┐
                  │ В ручном      │────Да────▶ Рейтинг = 0, прокси игнорируется
                  │ чёрном списке?│
                  └───────┬───────┘
                          │ Нет
                          ▼
                  ┌───────────────┐
                  │ IP в автомати-│────Да────▶ Рейтинг умножается ──▶ Остаётся в пуле
                  │ ческом списке?│           на 0.75 ^ hits
                  └───────┬───────┘
                          │ Нет
                          ▼
                Рейтинг сохранён ──▶ Остаётся в пуле
```

## 5. Периодическая переоценка (Health Check)

Раз в несколько минут все рабочие прокси проверяются заново. Цель — убедиться, что они всё ещё работают, и обновить задержку, скорость и успешность.

Если на очередной проверке HTTP-прокси перестал поддерживать CONNECT, он тоже отбрасывается. Прокси с пометкой `failed` не удаляются из базы сразу, но не попадают в топ-список и перепроверяются при следующем цикле.

```text
        Запуск health check
              │
              ▼
   Взять все рабочие прокси
              │
              ▼
   Проверить каждый заново
              │
              ▼
       ┌──────────────┐
       │ Проверка     │────Нет────▶ Отметить как нерабочий
       │ успешна?     │
       └──────┬───────┘
              │ Да
              │
              ▼
       ┌──────────────┐
       │ HTTP-прокси? │
       │ Есть CONNECT?│
       └──────┬───────┘
              │
   ┌──────────┴──────────┐
  Нет                 Да / SOCKS
   │                     │
   ▼                     ▼
Отбросить      Обновить статистику и рейтинг
                         │
                         ▼
                 Сохранить состояние
```

## 6. Формирование топ-списка

Прокси сортируются по рейтингу. Лучшие попадают в топ-список, который видит пользователь.

```text
 Все проверенные прокси
         │
         ▼
 Сортировка по рейтингу
         │
         ▼
  Топ-30 лучших
         │
         ▼
 Пользовательский выбор
 или авто-выбор
```

## 7. Использование прокси

Пользователь направляет свой трафик на локальный адрес `127.0.0.1:8080`. huntproxy принимает этот трафик и перенаправляет через выбранный upstream-прокси. Если выбранный прокси перестал отвечать, система переключается на следующий по рейтингу.

```text
   Пользователь открывает сайт
                │
                ▼
   ┌────────────────────────────┐
   │ Запрос на 127.0.0.1:8080   │
   └──────────────┬─────────────┘
                  │
                  ▼
   ┌────────────────────────────┐
   │ huntproxy выбирает активный│
   │ прокси                     │
   └──────────────┬─────────────┘
                  │
                  ▼
          ┌──────────────┐
          │ Прокси       │────Нет────▶ Переключиться на следующий прокси
          │ отвечает?    │               (возврат к выбору активного)
          └──────┬───────┘
                 │ Да
                 ▼
   Запрос уходит через прокси в интернет
                 │
                 ▼
   Ответ возвращается пользователю
```

---

## 8. Учёт трафика и событий

Пока работает локальный прокси, huntproxy записывает объём трафика (отдача/загрузка), запрашиваемые сайты и ошибки. Эти данные агрегируются по минутам и показываются на странице Analytics и в журнале событий.

```text
 Работа локального прокси
         │
         ▼
 Запись запросов и объёма трафика
         │
         ▼
 Агрегация по минутам
         │
         ▼
 Графики и логи в веб-интерфейсе
```

---

## 9. Управление чёрными списками

**Ручной чёрный список** — оператор добавляет адрес прокси. Такой прокси исключается из работы полностью.

**Автоматический чёрный список по IP** — система загружает списки IP-адресов (например, выходные узлы Tor). Если egress IP прокси совпадает с таким адресом, рейтинг прокси снижается, но он остаётся в пуле.

```text
                Прокси
                  │
                  ▼
        ┌───────────────────┐
        │ В ручном чёрном   │────Да────▶ Полностью исключён
        │ списке?           │
        └─────────┬─────────┘
                  │ Нет
                  ▼
        ┌────────────────────┐
        │ Egress IP в автома-│────Да────▶ Рейтинг снижен
        │ тическом списке?   │
        └─────────┬──────────┘
                  │ Нет
                  ▼
        Используется по рейтингу
```

## Роли блоков

| Блок | Что делает |
|------|------------|
| Загрузка списков | Берёт адреса прокси из внешних источников |
| Первичная проверка | Параллельно проверяет HTTP и SSL; HTTP-прокси должен поддерживать CONNECT; SOCKS — туннель; определяет страну и MITM |
| Замер скорости | HTTPS-замер при SSL + CONNECT, иначе HTTP-замер |
| Расчёт рейтинга | Оценивает прокси по успешности, задержке, скорости, SSL/CONNECT и MITM |
| Чёрные списки | Ручной — исключает; автоматический по IP — понижает рейтинг |
| Health check | Регулярно повторно проверяет рабочих; отбрасывает HTTP-прокси без CONNECT |
| Топ-список | Показывает лучшие прокси для выбора |
| Локальный прокси | Принимает трафик пользователя и направляет через выбранный прокси |
| Аналитика | Считает и визуализирует трафик, события и статистику |
