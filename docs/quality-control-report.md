# Отчёт: Контроль качества кода и архитектурные принципы

## Назначение

Документ описывает систему контроля качества кода для продуктов на стеке
Python (asyncio backend) + vanilla JS (static frontend) + SQLite
(persistence). Документ применим к текущему и новым продуктам на данном
стеке.

---

## 1. Архитектурные принципы

Контроль качества основан на принципах SOLID, адаптированных под
прагматику высокоскоростной разработки.

### Single Responsibility (S)

Каждый модуль, класс и функция отвечает за одну задачу. Слой диспетчеризации
не содержит бизнес-логики; слой планирования не содержит логики исполнения;
слой сохранения не содержит логики валидации.

**Контроль:** архитектурные тесты проверяют размеры файлов (порог 500 строк)
и цикломатическую сложность функций (порог 15). Превышение — задача на
рефакторинг, а не допустимое состояние.

### Open/Closed (O)

Новая функциональность добавляется через регистрацию, а не редактирование
существующих цепочек. Маршруты регистрируются в словаре роутера; типы задач
регистрируются в реестре исполнителя. Добавление нового элемента не требует
модификации диспетчера.

**Контроль:** тесты контракта проверяют, что диспетчер не содержит
хардкоженных ветвлений по типам. Добавление элемента = одна строка
регистрации.

### Liskov Substitution (L)

Подмена любой реализации с тем же интерфейсом не нарушает работу системы.
Тесты используют заглушки (stubs), заменяющие реальные исполнители через
единый интерфейс реестра. Если заглушка работает — любой совместимый
исполнитель будет работать.

### Interface Segregation (I)

Модули зависят от узкого набора атрибутов, а не от всего объекта состояния.
Планировщику нужны только флаги занятости и жизненный цикл задач; он не
обращается к доменным данным. Исполнитель обращается к доменным данным, но
не управляет очередью.

**Контроль:** тесты разделения проверяют, что планирование не касается
доменных данных, а исполнение не управляет планированием.

### Dependency Inversion (D)

Зависимости направлены от высокоуровневых модулей к низкоуровневым
абстракциям, не наоборот. Роутер — чистый leaf-модуль без зависимостей от
домена. Модели данных не импортируют ничего из проекта. Планировщик не
импортирует транспортный слой. Исполнитель не импортирует планировщик.

**Контроль:** тесты границ импортов проверяют отсутствие циклических
зависимостей и обратных направлений.

## 2. Система контроля качества

### 2.1 Четыре уровня тестов

| Уровень | Назначение | Когда запускать | Блокирует коммит |
|---------|-----------|-----------------|-----------------|
| **Функциональные** | Поведение системы корректно | Каждый коммит (pre-commit hook) | Да |
| **Контрактные** | Архитектурные границы не нарушены | Каждый коммит + отдельно | Да |
| **Архитектурные** | Структурные инварианты соблюдены | Отдельно (`--arch` / `--quality`) | Нет (backlog) |
| **Безопасности** | Уязвимости и некорректная обработка ввода | Отдельно (`--security`) | Нет (backlog) |

Архитектурные тесты **не блокируют** коммиты — они представляют backlog
рефакторинга. Каждый пройденный тест = одна решённая задача. Каждый
проваленный тест = конкретная задача на будущее. Пороги только снижаются,
никогда не повышаются.

Тесты безопасности аналогично не блокируют коммиты в локальной разработке,
но при переходе к CI/CD становятся блокирующими на уровне Pull Request.
Порог безопасности — всегда 0: одна уязвимость не может быть «допустимой».

### 2.2 Категории тестов

**Функциональные тесты** — проверяют поведение: API-эндпоинты возвращают
корректные данные, фоновые задачи запускаются и завершаются, бизнес-логика
работает корректно.

**Тесты контракта роутера** — каждый HTTP-эндпоинт отвечает (не 404). Safety
net для рефакторинга маршрутизации: если эндпоинт потерян при разбиении
модуля, тест немедленно падает.

**Тесты контракта исполнителя** — реестр исполнителей полон (каждый тип
задачи имеет обработчик), исполнители заменяемы через заглушки, ошибки и
отмены обрабатываются корректно, планирование не содержит логики
исполнения.

**Архитектурные тесты** — структурные инварианты:

- Размеры файлов не превышают пороги (снижаются по мере рефакторинга)
- Цикломатическая сложность функций не превышает 15 (проверка через ruff C901)
- Количество базовых классов композиции ограничено (God Object detection)
- Количество обращений к внутренностям объекта состояния ограничено
- Границы импортов соблюдены (нет циклов, нет обратных зависимостей)
- Модули-листья (роутер, модели) не импортируют домен
- Запрет на глушение ошибок: bare `except` (E722) и blind `except Exception`
  без re-raise (BLE001) — выявляются через ruff
- Покрытие веток (branch coverage) не падает ниже зафиксированного baseline

### 2.3 Режимы запуска

```
test.sh                # функциональные + контрактные (исключая slow и arch)
test.sh --arch         # только архитектурные инварианты
test.sh --router       # только контракт роутера
test.sh --executor     # только контракт исполнителя
test.sh --quality      # все архитектурные + контрактные вместе
test.sh --security     # тесты безопасности (SAST + SCA + фаззинг)
test.sh --coverage     # функциональные + отчёт покрытия веток
test.sh --all          # всё включая slow
test.sh -k <pattern>   # фильтр по имени (проброс в pytest)
```

### 2.4 Статический анализ frontend

ESLint запускается перед тестами, проверяет JavaScript на:

- Неопределённые переменные (`no-undef`)
- Неиспользуемые переменные (`no-unused-vars`)
- Опечатки и runtime-баги, которые Python-тесты не видят

0 ошибок — обязательно. Предупреждения — допустимы.

## 3. Инструменты

| Инструмент | Назначение | Уровень |
|------------|-----------|---------|
| **pytest** | Функциональное и контрактное тестирование | Ядро |
| **pytest-asyncio** | Тестирование async-кода | Ядро |
| **pytest markers** | Разделение тестов по категориям (arch, router, executor, fuzz, slow) | Организация |
| **ruff** | Линтинг Python + цикломатическая сложность (C901) + запрет silent-except (E722, BLE001) | Линтинг + метрики |
| **pytest-cov** | Покрытие кода с контролем веток (`--cov-branch`) | Покрытие |
| **bandit** | SAST: статический анализ безопасности Python (SQL injection, shell injection, weak crypto, hardcoded creds, unsafe deserialization) | Безопасность |
| **pip-audit** | SCA: проверка зависимостей на известные CVE (PyPA advisory database) | Безопасность |
| **hypothesis** | Фаззинг HTTP-эндпоинтов: path traversal, malformed input, oversized bodies, нестандартные методы | Безопасность |
| **ESLint** | Статический анализ JavaScript | Frontend |
| **AST-анализ (Python)** | Подсчёт обращений к атрибутам, God Object, coupling, silent-except detection | Архитектура |
| **pre-commit hook** | Автоматический запуск тестов перед коммитом | Git |

### 3.1 Почему ruff, а не radon + flake8 + pylint

Ruff заменяет три инструмента одним:

- **radon** — цикломатическая сложность (C901 в ruff)
- **flake8** — линтер (E722, F401, F841 в ruff)
- **pylint** — God Object detection (частично покрыто AST-тестами)

Преимущества: одна установка, одна конфигурация (`ruff.toml`), скорость
выполнения в ~100 раз выше, полная совместимость с современным Python
(match/case, walrus operator).

### 3.2 Запрет на глушение ошибок (AI-антипаттерн)

Главный AI-антипаттерн — оборачивание проблемного кода в `try/except: pass`
для формального прохождения тестов. Ruff правила:

- **E722** — bare `except:` без указания типа исключения
- **BLE001** — blind `except Exception` без re-raise или логирования

Эти правила проверяются в архитектурных тестах как backlog. Существующий
долг фиксируется, новый код не должен добавлять нарушений.

### 3.3 Контроль покрытия веток (branch coverage)

Покрытие линий (`--cov`) не ловит удаление логики: если `if` ветка удалена,
линия может остаться покрытой. Контроль веток (`--cov-branch`) ловит это —
удаление любой ветки снижает coverage.

Порог устанавливается по **текущему baseline** (не по идеалу) и повышается
монотонно. Попытка установить порог 90% на проекте с 57% покрытия
заблокирует все коммиты — порог должен быть достижим.

### 3.4 Отказ от контроля docstrings

Контроль наличия docstrings (инструменты вроде interrogate) уязвим перед
AI: агент генерирует сотни строк очевидных комментариев ("AI-slop") для
прохождения теста. Контроль качества документации переносится на этап
code review (Pull Request), где проверяется смысл, а не наличие текста.

### 3.5 Безопасность: SAST, SCA и фаззинг

Сетевой продукт принимает ввод из неконтролируемого источника (HTTP).
Любая точка приёма данных — потенциальная уязвимость. Контроль
безопасности разделён на три уровня, каждый из которых покрывает
вектор, недоступный двум другим.

**SAST (статический анализ)** — bandit сканирует исходный код на
известные классы уязвимостей: SQL-инъекции, shell-инъекции, слабую
криптографию, захардкоженные секреты, небезопасную десериализацию.
Ложные срабатывания подавляются комментарием `# nosec` с указанием
причины — подавление без причины запрещено. Тест проверяет отсутствие
находок HIGH и MEDIUM без обоснованного подавления.

**SCA (анализ зависимостей)** — pip-audit проверяет runtime-зависимости
против базы известных уязвимостей (PyPA advisory database). Проверяются
только зависимости из `requirements.txt`, dev-инструменты исключены —
уязвимость в pytest не попадает в production. Порог: 0 известных CVE.

**Фаззинг (динамический анализ)** — hypothesis генерирует
противоречивые и невалидные входы для HTTP-эндпоинтов: некорректные
query-параметры, мусор в JSON-теле, path traversal, превышенные размеры.
Принцип: **сервер не должен обрывать соединение или возвращать 500 на
произвольный ввод.** Корректные ответы — 200, 400 или 404. Обрыв
соединения означает необработанное исключение — это баг, не «особенность
обработки ошибок».

**Контроль:** тесты помечены маркером `fuzz`, запускаются через
`./test.sh --security`. SAST и SCA — архитектурные тесты (`arch`).
Фаззинг — отдельная категория (`fuzz`).

### 3.6 Валидация ввода на сетевой границе

Данные из HTTP-запроса (query-параметры, JSON-тело, сегменты пути) не
типизированы. Python не защищает от `int("abc")` или обращения к методу
`.get()` у результата `json.loads(b"0")` — последнее возвращает `int`, а
не `dict`. Каждая точка приёма должна валидировать тип **до** передачи в
бизнес-логику.

**Контроль:** хелперы `_int_param()` и `_json_body()` инкапсулируют
валидацию — невалидный ввод возвращает безопасное значение по умолчанию,
никогда не бросает исключение. Прямой вызов `int(qs.get())` или
`json.loads(body).get()` в handler'ах запрещён. Проверяется на этапе
фаззинга: некорректный ввод не должен вызывать необработанное
исключение.

## 4. Метрики качества

### 4.1 Цикломатическая сложность (Cyclomatic Complexity)

Измеряет количество независимых путей выполнения в функции. Проверяется
через ruff C901.

| Оценка | CC | Интерпретация |
|--------|-----|---------------|
| A | 1-5 | Простая, легко тестируется |
| B | 6-10 | Допустимая |
| C | 11-15 | На грани, кандидат на рефакторинг |
| D | 16-20 | Высокая, требует разбиения |
| E | 21-30 | Очень высокая |
| F | 31+ | Критическая, неприемлема |

**Порог:** CC ≤ 15 для всех функций. Порог 8 неприемлем для async-кода с
retry-циклами и обработкой таймаутов — функция с 3 if + 2 except + 1 for
уже достигает CC=7 искусственно.

### 4.2 Размер модуля

Файл более 500 строк — кандидат на разбиение. Порог регистрируется в
архитектурных тестах для каждого файла индивидуально. Новый файл свыше
500 строк должен быть зарегистрирован — иначе тест падает, заставляя
разработчика принять осознанное решение.

### 4.3 Coupling (связанность)

Количество обращений к внутренностям другого модуля. Высокая связанность
(>100 обращений из одного модуля) — признак God Object. Порог: ≤ 100
обращений.

### 4.4 God Object detection

Количество базовых классов в композиции и количество атрибутов в
`__init__`. Растущее число — сигнал что класс берёт на себя слишком много
ответственностей. Пороги фиксируются и снижаются по мере рефакторинга.

### 4.5 Покрытие веток (Branch Coverage)

Доля веток кода, покрытых тестами. В отличие от покрытия линий, ловит
удаление условной логики. Порог устанавливается по текущему baseline и
повышается монотонно — никогда не устанавливайте порог выше текущего
состояния.

### 4.6 Метрики безопасности

В отличие от архитектурных метрик, пороги безопасности **всегда 0** и не
имеют backlog. Одна SQL-инъекция — не «задача на будущее», а дыра.
Разница лишь в том, блокирует ли тест коммит: в локальной разработке —
нет, в CI/CD — да.

| Метрика | Порог | Метод |
|---------|-------|-------|
| Уязвимости SAST (HIGH/MEDIUM) | 0 | bandit, тест `TestBanditClean` |
| CVE в runtime-зависимостях | 0 | pip-audit, тест `TestNoKnownCVEs` |
| Обрыв соединения на malformed input | 0 | hypothesis, тесты `fuzz` |
| 500 на malformed input | 0 | hypothesis, тесты `fuzz` |

Подавление ложных срабатываний SAST требует комментария `# nosec` с
указанием причины. Подавление без причины приравнивается к нарушению.

## 5. Жизненный цикл контроля качества

### 5.1 Поток коммита

```
Разработчик вносит изменения
        │
        ▼
pre-commit hook запускает test.sh
        │
        ├── ESLint (frontend) ──── ошибки? ─── блокировка
        │
        ├── Ruff (Python) ─────── unused imports/vars? ── блокировка
        │
        ├── Функциональные тесты ── fail? ──── блокировка
        │
        ├── Контракт роутера ────── fail? ──── блокировка
        │
        ├── Контракт исполнителя ── fail? ──── блокировка
        │
        ├── Архитектурные тесты ─── исключены из блокировки
        │       (silent-except, complexity, coverage — backlog)
        │
        └── Тесты безопасности ──── исключены из блокировки
                (bandit, pip-audit, fuzz — run via `./test.sh --security`)
        │
        ▼
Коммит принят
```

### 5.2 Поток безопасности

```
Запуск --security ─── проваленный тест = уязвимость
        │
        ├── SAST: HIGH/MEDIUM? ──── фикс или # nosec с причиной
        │
        ├── SCA: CVE в deps? ────── обновить версию в requirements.txt
        │
        └── Фаззинг: 0 или 500? ──── хелпер валидации + обработка исключений
        │
        ▼
--security ─── тест проходит? ─── коммит
```

В отличие от потока рефакторинга, порог безопасности не снижается — он
всегда 0. Нельзя «временно допустить» одну уязвимость в backlog. Если
находка ложная — подавляется с причиной. Если реальная — фиксится.

### 5.3 Поток рефакторинга

```
Запуск --arch ─── проваленный тест = конкретная задача
        │
        ▼
Рефакторинг: разбиение модуля / функции / устранение silent-except
        │
        ▼
Функциональные тесты ──── fail? ──── откат, логика нарушена
        │
        ▼
Контрактные тесты ────── fail? ──── откат, границы нарушены
        │
        ▼
Снижение порога / повышение baseline в архитектурном тесте
        │
        ▼
--arch ─── тест проходит? ─── коммит
        │
        ▼
Порог зафиксирован — не может быть повышен без явного решения
```

### 5.4 Принцип однонаправленности порогов

Пороги метрик **только снижаются** (complexity, file size, coupling) или
**только повышаются** (coverage baseline), никогда не меняются
автоматически в обратную сторону. Изменение порога требует явного коммита
с обоснованием. Это гарантирует, что качество кода монотонно улучшается, а
не деградирует со временем.

## 6. Рекомендации для новых продуктов

1. **Внедрить архитектурные тесты с первого дня.** Пороги по текущему
   состоянию, затем снижать. Запускать отдельно от функциональных, чтобы
   не блокировать разработку, но держать backlog видимым.

2. **Роутер с registry, не if/elif.** С первого эндпоинта — регистрация в
   словаре. Это предотвращает рост мега-функции диспетчеризации.

3. **Разделить планирование и исполнение.** Планировщик (очередь, мьютексы,
   расписание) и исполнитель (бизнес-логика) — отдельные классы с
   registry-интерфейсом между ними.

4. **Разделять тесты по маркерам.** Функциональные, контрактные,
   архитектурные, slow — каждый уровень запускается независимо. Коммиты
   блокируют только функциональные + контрактные.

5. **Ruff с первого дня.** Одна установка заменяет radon + flake8 + pylint.
   Правила E722 (bare except) и BLE001 (blind except) — обязательны с
   первого коммита, не добавляются потом.

6. **Pre-commit hook обязателен.** Единственный способ гарантировать, что
   тесты реально запускаются перед коммитом, а не «потом».

7. **ESLint для frontend — даже если backend основной.** Python-тесты не
   видят JS-баги. ESLint ловит опечатки и undefined-переменные за секунды.

8. **Документировать пороги в тестах, не в Wiki.** Порог живёт рядом с
   тестом, который его проверяет. Разработчик видит текущее значение, цель
   и комментарий — без переключения контекста.

9. **Покрытие веток, не линий.** `--cov-branch` ловит удаление условной
   логики, которое `--cov` пропускает. Порог = текущее состояние, повышать
   монотонно.

10. **Не контролировать docstrings автоматически.** AI генерирует
    бессмысленные комментарии для прохождения теста. Контроль смысла
    документации — на code review, не на CI.

11. **SAST с первого дня.** Bandit запускается на каждом коммите. Порог:
    0 HIGH/MEDIUM без обоснованного `# nosec`. Добавить SAST позже —
    значит расхлёбывать накопленный долг вместо предотвращения.

12. **SCA на релизах.** pip-audit проверяет runtime-зависимости на CVE
    перед каждым релизом. Новая зависимость — проверка на CVE до
    добавления в `requirements.txt`. Runtime и dev-зависимости
    разделены.

13. **Фаззинг сетевой границы.** Каждый HTTP-эндпоинт фаззится
    hypothesis: некорректные query-параметры, мусор в JSON-теле, path
    traversal, превышенные размеры. Ни одного обрыва соединения или 500
    на произвольный ввод. Фаззинг ловит то, что SAST не видит — ошибки
    на стыке «данные пришли → парсятся → используются».

14. **Валидация ввода через хелперы, не через ad-hoc парсинг.** Прямой
    `int(qs.get())` и `json.loads(body).get()` в handler'ах запрещены.
    Хелпер гарантирует: невалидный ввод → безопасное значение, никогда
    исключение. Это не удобство, а security-контроль.

15. **Лимиты ресурсов на transport-уровне.** Таймаут на чтение заголовков
    и тела, лимит размера тела, лимит длины URL. Без лимитов один клиент
    исчерпает память или заблокирует event loop.

## 7. Антипаттерны, которые система обнаруживает

| Антипаттерн | Как обнаруживается | Инструмент |
|-------------|-------------------|------------|
| God Object | Тест количества базовых классов и атрибутов | AST-анализ |
| Spaghetti-функция | Тест цикломатической сложности | ruff C901 |
| Megafile | Тест размера файла | AST-анализ |
| Tight coupling | Тест количества обращений к атрибутам | AST-анализ |
| Circular dependency | Тест границ импортов | AST-анализ |
| Monolithic dispatcher | Тест количества ветвлений в диспетчере | AST-анализ |
| Silent except (AI #1) | ruff E722 + BLE001 | ruff |
| Bare except | ruff E722 | ruff |
| Blind catch-all | ruff BLE001 | ruff |
| Stale flags | Функциональные тесты | pytest |
| Lost endpoints | Контракт роутера | pytest |
| Silent task drop | Контракт исполнителя | pytest |
| Deleted logic | Branch coverage drop | pytest-cov |
| Unused imports/vars | ruff F401 + F841 | ruff |
| AI-slop docstrings | Code review (не автоматизировано) | ручная проверка |
| SQL injection | B608 — string-based query construction | bandit |
| Shell injection | B602/B603 — subprocess shell=True | bandit |
| Weak crypto | B324 — insecure hash/random | bandit |
| Hardcoded credentials | B105 — passwords in source | bandit |
| Unsafe deserialization | B301 — pickle/yaml.load | bandit |
| Known CVEs in deps | PyPA advisory database | pip-audit |
| Path traversal | `../` in static file paths | hypothesis fuzz |
| Unhandled int() on input | Malformed query params → server crash | hypothesis fuzz |
| Non-dict JSON body | `json.loads(b'0').get()` → AttributeError | hypothesis fuzz |
| Oversized body DoS | Large Content-Length → hang | hypothesis fuzz |
